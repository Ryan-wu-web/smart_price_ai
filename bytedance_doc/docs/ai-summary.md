## 二、AI 任务编排体系

### 2.1 任务分层模型

项目将 AI 能力划分为三个层次，每个层次对应不同的模型调用策略和 Prompt 设计模式：

| 层级 | 任务类型 | 调用模型 | Prompt 策略 |
|------|---------|---------|------------|
| **L1 感知层** | 图像识别、属性提取 | Doubao-VLM | 结构化约束 + Few-shot 示例 |
| **L2 认知层** | 对话生成、意图识别 | Doubao-LLM | 角色设定 + 上下文注入 + Action 路由 |
| **L3 决策层** | 报告生成、趋势分析 | Doubao-LLM | 数据驱动 + 模板填充 + 多维度评估 |

---

### 2.2 L1 感知层：商品识别 Prompt

**场景**：用户拍摄商品照片，需要提取品牌、品类、颜色、款式等结构化信息。

**核心挑战**：
- 模型可能"猜测"品牌（将未知品牌硬猜为知名品牌）
- 输出格式不稳定（有时加解释文字，有时纯 JSON）
- 品类描述过于笼统（如"物品"而非"运动鞋"）

**Prompt 设计策略**：

```python
"你是一位专业的商品识别专家。请根据以下图片描述，提取结构化信息并以 JSON 格式输出。\n\n"
"输出字段：name（商品名称）、brand（品牌，无法识别则为空字符串）、"
"category（品类）、color（主色调）、material（材质）、style（款式）。\n\n"
"约束：\n"
"- 如果无法识别品牌，brand 设为空字符串，不要猜测\n"
"- category 必须是具体品类，如运动鞋、手机、口红、零食等\n"
"- color 用简洁的中文，如纯白色、黑色、红色\n"
"- material 无法识别设为空字符串\n"
"- style 描述具体款式，如低帮板鞋、连衣裙、无线耳机\n\n"
"示例1（完整识别）：\n"
'{"name": "Nike Air Force 1 白色 42码", "brand": "Nike", "category": "运动鞋", ...}\n\n'
"示例2（部分未知）：\n"
'{"name": "白色运动鞋", "brand": "", "category": "运动鞋", ...}\n\n'
"请只输出 JSON，不要添加任何解释文字。"
```

**设计要点**：
- **角色设定**："专业的商品识别专家"，约束模型输出的专业性和准确性
- **字段级约束**：每个字段都有明确的规则，特别是 `brand` 的"不猜测"原则
- **Few-shot 示例**：提供完整识别和部分未知两种场景，引导模型正确处理边界情况
- **输出格式锁定**：最后一句"只输出 JSON"强制纯结构化输出

**鲁棒性保障**：
```python
# JSON 解析失败时的容错回退
try:
    parsed = json.loads(response)
except json.JSONDecodeError:
    # 正则提取：从非标准输出中抢救关键字段
    match = re.search(r'"brand"\s*:\s*"([^"]*)"', response)
    brand = match.group(1) if match else ""
```

### 2.3 L2 认知层：AI 导购对话 Prompt

**场景**：用户与 AI 进行多轮对话，AI 需要理解意图、记住上下文、生成决策卡片。

**核心挑战**：
- 意图识别（用户想对比、想买、还是问价格走势？）
- 上下文记忆（多轮对话中保持商品信息不丢失）
- Action 路由（根据意图触发不同的决策卡片类型）

**Prompt 设计策略**：

```python
"你是购物顾问'小价'，语气亲切自然，适当使用 emoji。\n\n"
"当前关注商品：Nike Air Force 1（¥799）\n\n"
"最近对话：\n"
"用户：帮我选一双运动鞋\n"
"助手：推荐 AF1 和 Dunk...\n\n"
"用户：对比一下 AF1 和 Dunk\n\n"
"以 JSON 输出：reply, action, action_data, current_product。\n\n"
"action 规则：\n"
"- 含'对比/比较/vs' → report + report_type=comparison\n"
"- 含'购买建议/什么时候买/在哪买' → report + report_type=buy_guide\n"
"- 含'哪个好/推荐/帮我选' → report + report_type=decision\n"
"- 含'走势/历史价格/趋势' → trend\n"
"- 其他 → none\n\n"
"action_data 格式：\n"
"comparison: {...}\n"
"buy_guide: {...}\n"
"decision: {...}\n\n"
"示例1（对比）：{...}\n"
"示例2（购买建议）：{...}\n"
"只输出 JSON，不要解释。"
```

**设计要点**：
- **角色人格化**："小价" + emoji，降低对话机械感
- **上下文注入**：将最近 6 轮对话和当前商品信息注入 Prompt，实现"记忆"
- **规则引擎**：用明确的规则替代模糊的意图判断，确保 Action 类型准确
- **Schema 示例**：每种 Action 都附带完整的 JSON 示例，降低模型出错的概率

**多目标识别扩展**：

在单目标识别 Prompt 基础上，增加多目标检测能力：

```python
"识别图片中的所有商品，返回 JSON 数组。"
"每个元素包含：name, brand, category, color, center{x,y}"
"center 为归一化坐标（0-1），表示商品在图片中的中心位置。"
```

这是项目的核心创新点之一 — 通过 Prompt 扩展让同一模型支持从"单目标识别"到"多目标检测"的能力跃迁。

### 2.4 L3 决策层：报告生成 Prompt

**场景**：基于对话上下文和比价数据，生成结构化购买决策报告。

**Prompt 设计**：

```python
"你是一位购物决策顾问。根据以下信息，为用户生成一份购买决策报告。\n\n"
"目标商品：Nike Air Force 1\n"
"最佳选择：Nike Air Force 1（京东），¥749，评分 4.8\n"
"备选方案：\n"
"- Nike Dunk Low（天猫）：¥699，评分 4.7\n\n"
"请以 JSON 格式输出，包含字段："
"summary（摘要）、pros（优点列表）、cons（缺点列表）、recommendation（最终推荐语）。"
```

**设计要点**：
- **数据驱动**：将结构化数据（价格、评分、平台）直接注入 Prompt，避免模型"幻觉"
- **多维度评估**：从价格、评分、平台信誉等维度综合评估，而非单一指标
- **可解释性**：`pros` 和 `cons` 列表让用户理解决策依据，提升信任度

### 2.5 Prompt 工程方法论总结

| 原则 | 实践 | 效果 |
|------|------|------|
| **角色设定** | 为每个任务定义专业角色 | 提升输出专业性和一致性 |
| **结构化约束** | 明确字段、类型、格式规则 | 降低解析失败率 |
| **Few-shot 示例** | 提供边界场景示例 | 引导模型正确处理异常情况 |
| **上下文注入** | 将业务数据动态注入 Prompt | 实现个性化、有记忆的对话 |
| **规则引擎** | 用明确规则替代模糊判断 | 确保 Action 路由准确率 |
| **容错回退** | JSON 解析失败时正则提取 | 服务不中断，用户体验连续 |

### 3.2 Superpowers 方法论

项目采用 **Superpowers** AI 编程最佳实践框架，将开发流程标准化为五个阶段：

```
Brainstorming → Writing Plans → Executing Plans → Code Review → Finish Branch
```

#### 阶段 1：Brainstorming（构思设计）

**实践**：在编写任何代码前，先通过苏格拉底式提问明确需求。

**案例**：多目标识别 UI 美化
- 问：最需要改进的方向是什么？→ 视觉质感
- 问：什么风格？→ 精致卡片（Premium Card）
- 问：信息内容？→ 保持简洁（序号+名称）
- 问：动画调整？→ 优化缓动和时长

**产出**：`docs/superpowers/specs/2025-06-06-multi-object-ui-design.md`

#### 阶段 2：Writing Plans（编写计划）

**实践**：将设计拆分为 2-5 分钟的小任务，每个任务包含精确的文件路径和代码。

**案例**：UI 美化实现计划包含 7 个任务：
1. 气泡标签样式调整
2. 序号圆点配色反转
3. 连接线改为实线
4. 锚点改为空心圆环
5. Vignette 暗角加深
6. 动画参数优化
7. 锚点脉冲动画

**产出**：`docs/superpowers/plans/2025-06-06-multi-object-ui-plan.md`

#### 阶段 3：Executing Plans（执行计划）

**实践**：逐个任务执行，每步 `flutter analyze` + `git commit`。

**关键纪律**：
- 每个任务独立提交（7 个任务 = 7 个 commit）
- 每步运行静态检查，确保 0 issues
- 不跳过任何步骤，稳扎稳打

#### 阶段 4：Code Review（代码审查）

**实践**：对照计划逐项审查，按严重性分类问题。

**审查维度**：
- 功能正确性：是否满足规格？
- 代码质量：清晰可读？DRY？
- 架构一致性：遵循现有模式？

**案例**：审查发现三角形箭头颜色可微调以匹配新气泡风格（轻微问题，可选修复）。

#### 阶段 5：Finish Branch（完成分支）

**实践**：验证测试通过，合并代码，清理工作区。

### 3.3 上下文管理策略

AI 辅助编码的核心挑战是**上下文管理** — 如何让 AI 理解项目全貌而不超出 token 限制。

**策略 1：AGENTS.md 分层治理**

```
项目根目录 AGENTS.md          → 全局规则（技术栈、提交规范）
backend/AGENTS.md              → 后端专项（Python 规范、API 设计）
android-app/AGENTS.md          → 前端专项（Flutter 规范、UI 规范）
```

**策略 2：Skill 化知识沉淀**

将重复使用的知识封装为 Skill：
- `frontend-design`：前端设计规范、Design Token 体系
- `superpowers-using`：开发流程方法论
- `lark-*`：飞书生态工具链

**策略 3：Compation 机制**

长会话中自动压缩历史上下文，保留关键决策和代码状态，确保 AI 始终掌握项目全貌。

---

### 3.4 代码幻觉调优纠偏

AI 编码过程中常见的幻觉及应对策略：

| 幻觉类型 | 案例 | 纠偏策略 |
|---------|------|---------|
| **虚构 API** | AI 生成不存在的 `Image.compress()` 方法 | 要求 AI 先查阅现有代码库，确认 API 存在性 |
| **过度设计** | AI 建议引入 BLoC/Riverpod 状态管理 | 坚持 YAGNI 原则，当前规模 `setState` 足够 |
| **忽略约束** | AI 忽略"禁用端侧 AI SDK"的合规要求 | 在 AGENTS.md 中明确约束，每次任务前重申 |
| **样式不一致** | AI 使用硬编码颜色而非 Design Token | 要求 AI 查阅 `constants.dart`，强制使用 Token |

**最佳实践**：
- 每次代码修改前，要求 AI 先 `ReadFile` 查阅相关文件
- 复杂修改先写计划，获得批准后再执行
- 关键代码（如 Prompt 设计）人工审查后再提交

## 四、总结

Smart Price 项目的 AI 实践体现了两个核心能力：

1. **AI 任务编排能力**：将复杂业务需求（拍照识物、智能导购、决策报告）拆解为可编排的 AI 任务链，通过精心设计的 Prompt 实现稳定的结构化输出，并建立完善的容错机制保障服务连续性。

2. **AI 辅助编码能力**：建立系统化的协作流程（Superpowers 方法论），通过分层上下文管理、Skill 化知识沉淀、严格的代码审查机制，将 AI 从"代码生成器"提升为"协作工程师"。

这两者的结合，使得单人开发者能够在 20 天内完成一款功能完整、体验流畅的 AI 购物助手，充分展现了 AI 工具在提升研发效率方面的巨大潜力。
