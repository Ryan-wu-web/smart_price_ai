# Day 6：工程优化 — 为什么项目这么快？

> 目标：理解 dHash 缓存、连接池复用、图片压缩这三个核心优化的原理。

---

## 1. 缓存优化 — dHash 感知哈希

**问题**：同一商品拍两次，每次都要调 AI（8-10 秒），太慢了！

**方案**：缓存第一次的识别结果，第二次直接返回。

**但怎么判断"同一商品"？**

| 方案 | 问题 |
|------|------|
| 文件 MD5 | 重新拍一张照片，角度变了，MD5 完全不同 |
| 文件名 | 没有意义 |
| **dHash 感知哈希** ✅ | 容忍角度、亮度、轻微裁剪变化 |

**dHash 原理（简化版）**：
1. 把图片缩成 9×8 像素的灰度图（很小，计算快）
2. 比较相邻像素的亮度：左边比右边亮 → 1，暗 → 0
3. 得到 64 位二进制字符串（如 `10110011...`）
4. 这个字符串就是图片的"指纹"

**为什么能容忍重新拍照？**
- 图片缩到 9×8 后，细节丢失，只保留整体结构
- 角度变化一点、亮度变化一点，指纹变化不大
- 汉明距离（指纹差异度）< 10 就认为"同一商品"

**效果**：
- 首次识别：~8-10 秒（调 AI）
- 二次识别：**< 0.1 秒**（缓存命中）

## 2. 连接池复用

**问题**：每次调 AI 都要新建 HTTP 连接（TCP 握手 0.5-1 秒），浪费！

**方案**：连接池复用。

**原理**：
- 第一次：新建连接 → 调 AI → 不关闭，留着
- 第二次：直接用上次留下的连接 → 省掉握手时间

**类比**：餐厅不是每道菜都现雇一个服务员，而是固定几个服务员循环服务。

**代码实现**：
```python
class BaseAPIClient:
    def __init__(self):
        self._client = httpx.AsyncClient()  # 实例级持有，复用
    
    async def _post(self, ...):
        return await self._client.post(...)  # 复用同一个 client
```

## 3. 图片压缩

**问题**：手机拍的照片 3-5MB，传给后端太慢。

**方案**：压缩后再传。

**我们的压缩策略**：
- 缩放到 600px 宽（保持比例）
- JPEG 质量 75%（人眼几乎看不出损失）
- 转 RGB（去掉透明通道）

**效果**：图片从 3MB 降到 ~200KB，传输时间减少 30-40%。

---

## 4. 三个优化的关系

```
用户拍照
  ↓
图片压缩（省传输时间）
  ↓
算 dHash（省计算时间）
  ↓
查缓存？
  → 有：直接返回（< 0.1 秒）
  → 没有：调 AI（连接池复用，省握手时间）
```
