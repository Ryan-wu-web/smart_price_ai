# Day 1：项目全景 — 先看到森林，再看到树木

> 目标：理解 Smart Price 的整体架构、数据流、技术选型理由。看完你能向陌生人讲清楚"这个项目是干嘛的、怎么做的"。

---

## 1. 这个项目解决什么问题？

**一句话**：帮消费者用 AI 识别商品，然后比价、导购，最终做出购买决策。

**场景**：你在商场看到一双鞋，不确定值不值得买。打开 Smart Price → 拍照 → AI 识别品牌和型号 → 看各平台价格 → 问 AI "什么时候买最划算" → 得到决策报告。

---

## 2. 系统长什么样？（四层架构）

想象一个餐厅：

| 层级 | 类比 | 职责 | 我们用的技术 |
|------|------|------|------------|
| **客户端** | 顾客 + 服务员 | 接待用户、展示菜单、传菜 | Flutter App |
| **服务端** | 后厨 | 做菜、协调各工序 | FastAPI |
| **AI 引擎** | 特聘大厨 | 识别食材、推荐菜品 | 火山引擎 Doubao |
| **数据层** | 冰箱 + 食材库 | 存食材、备料 | SQLite + Mock 数据 |

**数据怎么流动？**

```
用户拍照 → 客户端压缩图片 → 传给服务端
→ 服务端查缓存（有没有拍过？）
→ 没拍过 → 调用 AI 识别（VLM 视觉模型）
→ 返回结果 → 展示给用户
→ 用户问 AI → 调用 AI 对话（LLM 语言模型）
→ 生成决策建议
```

## 3. 为什么选这些技术？（评委最爱问）

### 为什么前端用 Flutter？

- **跨平台**：一套代码同时跑 Android 和 iOS，省时间
- **自绘能力强**：CustomPainter 可以画出气泡标签、扫描线等复杂效果
- **热重载**：改代码后秒看效果，开发效率高

**不是 React Native 吗？** RN 也可以，但 Flutter 的绘图能力更强，做自定义动画更方便。

### 为什么后端用 FastAPI？

- **异步原生**：Python 的 `async/await` 支持高并发，适合同时处理多个请求
- **自动文档**：启动后访问 `/docs` 就有 API 文档，不用手写
- **类型安全**：Pydantic 自动校验请求参数，减少 bug

**不是 Django/Flask 吗？** Django 太重，Flask 没有异步支持。FastAPI 是轻量 + 异步的最佳选择。

### 为什么 AI 用火山引擎 Doubao？

- **VLM + LLM 统一**：同一家厂商，视觉识别和语言对话都用它，对接成本低
- **中文优化**：对中文商品名识别准确率高
- **流式支持**：原生支持 SSE 流式输出，打字机效果容易实现

## 4. 项目的核心数据流（三条主线）

### 主线 A：拍照识物（单目标）

```
拍照 → 压缩（600px）→ 算 dHash → 查缓存 → 有？直接返回
                                          → 没有？调用 VLM → 存缓存 → 返回
```

### 主线 B：AI 导购聊天

```
发消息 → 加载历史对话 → 调用 LLM → 逐字返回（SSE）→ 前端逐字显示
→ 对话结束 → 生成决策卡片（对比/指南/报告）
```

### 主线 C：多目标识别

```
拍照 → 调用 VLM（多目标模式）→ 返回多个商品 + 坐标
→ 前端布局算法 → 气泡标签定位 → 动画弹出
```

---

## 5. 答辩话术（如果被问"讲讲你的项目架构"）

> "Smart Price 采用前后端分离的四层架构。前端用 Flutter 做跨平台 UI，核心功能是相机拍摄和气泡标签可视化。后端用 FastAPI 提供 RESTful API，负责业务编排。AI 层调用火山引擎的 Doubao 模型，分别用 VLM 做图像识别、LLM 做对话生成。数据层用 SQLite 做缓存，Mock 数据模拟多平台比价。整个系统的特点是所有 AI 计算在云端完成，前端只做展示和交互。"
